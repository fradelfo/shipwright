# Plan: sw-audit — Shipwright for Existing Projects

## Overview

Add `sw-audit` — a new Shipwright skill that helps users understand, triage, and rescue existing messy or inherited codebases. It scans the project inline (Architect-led), asks targeted questions about what code alone can't reveal, produces a structured audit artifact with a prioritised triage roadmap, and hands each roadmap item to the existing Plan → Build → Ship flow. No new agents needed; no existing phase skills modified.

## Codebase Context

- **Skills:** `skills/<name>/SKILL.md` — YAML frontmatter (`name`, `description`, `argument-hint`, `allowed-tools`), imperative body, under 500 lines, markdown links for file references
- **Agents:** `agents/<category>/<name>.md` — `type: agent`, `perspective`, no `description`, 40–60 lines — **not touched by this plan**
- **Roles:** `skills/roles/<name>.md` — passive definitions, referenced by skills via markdown links — **not touched by this plan**
- **Artifacts:** `docs/shipwright/<phase>/YYYY-MM-DD-<topic>.md` with YAML frontmatter (`type`, `topic`, `tier`, `status`, `date`, `upstream`)
- **sw-orchestrate extension point:** Tier Classification table + Routing section — straightforward to add a Rescue tier row and section
- **AGENTS.md:** Tracks directory structure; must reflect any new skill directory
- **plugin.json:** Version bump on each shipped phase

## Architecture Decisions

### ADR: Inline scan vs. dispatched codebase-analyst

**Status:** Accepted

**Context:** sw-plan dispatches `codebase-analyst` as a pre-flight research agent before design. sw-audit also needs to scan the codebase — the question is whether to reuse the same agent or scan inline.

**Options:**
- Dispatch codebase-analyst: consistent with sw-plan; but the agent is scoped for pattern inventory, not health assessment — extending it creates a dual-purpose agent with two output formats
- Inline scan: Architect role reads files directly as part of the skill flow; simpler; keeps codebase-analyst focused

**Decision:** Inline scan. The audit scan *is* the core work of the skill, not a pre-flight for design decisions. Keeping it inline makes sw-audit self-contained and leaves codebase-analyst unchanged.

**Consequences:** No agent files are created or modified. sw-audit's `allowed-tools` must include Read, Glob, Grep, Bash(git log *), Bash(find *), Bash(ls *).

---

### ADR: Scan depth heuristic

**Status:** Accepted

**Context:** Discovery flagged scan depth on large projects as an unknown. Three options considered.

**Options:**
- Always shallow (top-level + entry points only): fast but imprecise for small projects
- User-configurable: adds friction; users shouldn't have to know what "shallow" means
- Heuristic based on file count: natural, no friction

**Decision:** Heuristic — deep scan if project has <200 files, shallow (top-level + entry points + manifests) if larger. File count is a single Bash call. Uninspected areas are named explicitly in the audit artifact.

**Consequences:** Skill must count files before deciding depth. Large-project audits will explicitly list uninspected areas in the Health Signals section.

---

### ADR: Rescue tier in sw-orchestrate

**Status:** Accepted

**Context:** sw-orchestrate currently routes Quick / Standard / Major. Rescue signals ("inherited", "messy", "legacy", "chaos") don't fit any existing tier — Major assumes vague but new requirements, not existing disorder.

**Options:**
- Fourth Rescue tier: unambiguous routing; clean classification table
- Rescue as Major variant: simpler but conflates two different starting points

**Decision:** Add Rescue as a fourth tier in the classification table and routing section.

**Consequences:** sw-orchestrate gains a Rescue detection block and routing section. Full Lifecycle Reference table gains an Audit row.

## Risk Flags

- **SKILL.md length:** sw-audit is feature-rich (scan + interview + triage + artifact + handoff). Monitor line count during authoring — 500 line limit per AGENTS.md. Cut prose, not substance.
- **Rescue vs. Major signal overlap:** Phrases like "I'm not sure what to build" could be Major or Rescue. Resolution in sw-orchestrate: trust explicit rescue signals ("inherited", "legacy", "someone else wrote") over ambiguous ones; default ambiguous to Major.
- **Interview question count:** Discovery scoped to ≤5 questions. Enforce this in the skill — the scan must prioritise which ambiguities matter most, not ask about everything uncertain.
