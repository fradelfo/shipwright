# Discovery: sw-audit — Shipwright for Existing Projects

## Problem Statement

Shipwright currently assumes a greenfield starting point — you know what you're building and the codebase has patterns worth following. But the majority of real projects don't look like that. They're inherited, half-finished, or organically grown into chaos. For these projects, the existing flow breaks down immediately: you can't discover what to build, plan a task sequence, or follow conventions when you don't know what you have yet. `sw-audit` fills this gap: a pre-discovery phase that maps the existing project, surfaces debt and disorder, and produces a prioritised triage roadmap that feeds directly into the normal Shipwright flow.

## Success Criteria

- A user with an inherited or chaotic codebase can run `sw-audit` and receive a structured, actionable map of what they have within a single session
- The audit artifact is precise enough that any item on the triage roadmap can be passed directly to `/sw-plan` without additional context-gathering
- `sw-orchestrate` routes audit-signal phrases ("inherited", "messy", "don't know where to start") to `sw-audit` automatically
- The normal Discover → Plan → Build → Ship flow is unchanged; `sw-audit` is purely additive as a pre-entry point

## Non-Goals

- **Security vulnerability scanning** — CVE and dependency auditing belongs to dedicated tools (Snyk, Dependabot, GitHub security alerts); Shipwright is a methodology, not a security scanner
- **CI/CD health checks** — too project-specific to automate generically
- **New role (Tech Lead)** — existing Architect + PM roles cover the audit adequately; a new role adds weight without clear benefit at this stage
- **New `project-auditor` agent** — the existing `codebase-analyst` agent will be extended to produce the deeper health-assessment output needed; a new agent file is premature
- **Auto-running `/sw-plan`** — after the audit, the user decides what to tackle first; Shipwright suggests but never acts autonomously

## User Flows

### Happy Path — Inherited Codebase

1. User runs `/sw-audit` or `/sw-orchestrate "I just inherited this codebase"`
2. Architect role scans the project: directory structure, key files, git history, package manifest, test coverage signal, README age, framework detection
3. Shipwright identifies what it cannot infer from code alone and asks targeted questions — one at a time:
   - *"I see a `legacy/` folder with no commits in 18 months — is this intentionally frozen or abandoned mid-way?"*
   - *"There are two auth systems (`sessions/` and `jwt/`) — which is the intended direction?"*
   - *"Test coverage appears near zero — is testing a goal for this project?"*
4. PM role triages and prioritises findings: groups by category (cleanup / stabilisation / infrastructure / new feature), ranks by impact and effort
5. Audit artifact saved to `docs/shipwright/audit/YYYY-MM-DD-<project>.md`
6. Shipwright presents the triage roadmap, highlights the suggested first item, and offers: *"Run `/sw-plan docs/shipwright/audit/…` to start, or pick any item from the roadmap."*
7. User picks an item — normal Plan → Build → Ship flow takes over from here

### Happy Path — Active Chaos (Work In Progress Project)

1. User runs `/sw-audit "the project is mid-refactor and nothing is consistent"`
2. Scan focuses on: duplicate patterns, half-completed migrations, inconsistent naming, features with no tests, TODO/FIXME density
3. Interview focuses on intent: *"Which of these two approaches for X is the one you want to keep?"*
4. Roadmap is ordered by "restore consistency first, then ship" — stabilisation tasks ranked above new features
5. Handoff identical to happy path above

### Error Paths

1. **Minimal metadata** (no git history, no README, no manifest): Shipwright flags *"This project has minimal metadata — the audit will be less precise. I'll ask more questions to compensate."* Interview expands to cover what the scan can't infer.
2. **User doesn't know the answer**: *"I have no idea what the legacy folder is."* → Shipwright marks that area `status: unknown` in the audit artifact and moves on. Never blocks.
3. **Very large project** (>500 files): Shipwright scans entry points and top-level structure; flags uninspected areas explicitly in the audit artifact. Does not attempt to read every file.

## Screen Inventory

| Interaction | Purpose | Entry | Exit |
|-------------|---------|-------|------|
| `/sw-audit` invocation | Trigger the audit | User command | Scan begins |
| Targeted interview | Resolve ambiguities the scan cannot answer | After scan completes | After ≤5 questions |
| Audit artifact | Persistent map + roadmap; upstream for all follow-on plan artifacts | Written to disk | Referenced by `/sw-plan` |
| Roadmap presentation | Show triage list, suggest first item, offer handoff | After artifact saved | User picks item or exits |

## Feasibility Assessment

**Complexity:** Moderate

**Blockers:** None identified — `sw-audit` is a new skill file plus minor additions to `sw-orchestrate`. No new infrastructure, no new agents needed for MVP.

**Unknowns:**
- **Scan depth on large projects**: needs a policy decision — always top-level + entry points, or heuristic-based (e.g. scan deeply if <200 files, shallowly if larger). Resolve during planning.
- **`codebase-analyst` output format**: the existing agent produces a Pattern Inventory; `sw-audit` needs a Health Assessment output. Decide whether to extend the agent file or pass different instructions to the same agent. Resolve during planning.

## Scoped MVP Feature List

| In | Out | Reason for exclusion |
|----|-----|---------------------|
| `sw-audit` skill (new) | New Tech Lead role | Architect + PM cover the audit adequately |
| Code scan: directory structure, manifests, git log, README, framework detection | Security vulnerability scanning | Out of scope for a methodology tool |
| Targeted interview (≤5 questions, one at a time) | Automated CI/CD health check | Too project-specific |
| Audit artifact: project map + health signals + debt hotspots + triage roadmap | Dependency CVE scanning | Dedicated tool territory |
| `sw-orchestrate` routing triggers for audit-signal phrases | New `project-auditor` agent | Extend `codebase-analyst` for now |
| Handoff to `/sw-plan` with suggested first item | Auto-run plan without user confirmation | User decides, always |
| `codebase-analyst` extended for health-assessment output | `sw-audit --refresh` diff mode | Phase 2 |

## Phase 2 Candidates

- **`sw-audit --refresh`** — re-run the audit after a cleanup sprint and diff the delta against the previous audit artifact; shows measurable progress over time
- **Dedicated `project-auditor` agent** — a standalone agent with deeper health-assessment heuristics (cyclomatic complexity, file size histograms, duplicate detection) replacing the extended `codebase-analyst`
- **Tech Lead role** — a new role that bridges PM and Architect specifically for rescue projects; different prioritisation heuristics (stability before features, consistency before performance)
- **Complexity metrics** — if project tooling provides coverage reports or complexity scores, surface them in the audit artifact automatically
- **Audit index** — a `docs/shipwright/audit/index.md` tracking multiple audit runs over time for long-running rescue projects
