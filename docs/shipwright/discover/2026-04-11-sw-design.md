# Discovery: sw-design — UX Design Strategy Phase

## Problem Statement

The current UX Designer role in Shipwright is shallow: it maps a happy path and lists screens, then steps aside. Real design strategy — understanding the job the user is hiring the feature to do, the mental model they bring, the information architecture, and the interaction principles that should govern the experience — never happens. This thinking either gets collapsed into the PM's problem framing or skipped entirely. The result: features that work correctly but confuse users, because nobody asked "how will a real person think about this?"

`sw-design` adds a first-class design strategy phase with a UX Design Strategist lead role. It works both as a lifecycle phase (between Discover and Plan) and as an on-demand skill for any design question.

## Success Criteria

- A user can invoke `/sw-design` after discovery and receive a structured design strategy artifact before planning begins
- The design artifact is specific enough that the Architect can read it and understand what the user needs to *experience*, not just what to build
- `/sw-design` can also be invoked standalone — without upstream discovery — and produce a useful output on its own
- The UX Designer role is upgraded to a full UX Design Strategist with depth across JTBD, mental models, IA, interaction principles, design critique, and accessibility
- Both text-based artifacts and Mermaid diagrams are used where appropriate — text by default, Mermaid for complex IA hierarchies and branching decision flows

## Non-Goals

- **Visual design / mockups** — Shipwright produces text artifacts, not images or Figma exports
- **Usability testing** — out of scope; design strategy informs what to build, not validate what was built
- **Design system management** — component inventory is in scope; managing a design system is not
- **Replacing sw-discover** — sw-design is downstream of discovery, not a replacement; the PM still owns problem framing

## User Flows

### Happy Path — Phase Mode (after sw-discover)

1. User runs `/sw-design docs/shipwright/discover/2026-04-11-assign-task.md`
2. Skill reads the discovery doc — extracts problem statement, user flows, scoped MVP
3. UX Design Strategist applies JTBD framing: what job is the user hiring this feature to do?
4. Mental model mapping: how does the user think about this domain before touching the product?
5. Information architecture: how is the feature organised? Mermaid diagram if 3+ levels of hierarchy
6. User decision points: where does the user make decisions, what do they need at each? Mermaid if branching
7. Interaction principles: 3–5 named principles that govern the experience
8. Design critique: adversarial pass — where will users get confused even when the feature works correctly?
9. Accessibility strategy: how does this work for everyone?
10. Component/pattern inventory: what UI patterns apply, what needs to be invented?
11. Design artifact saved to `docs/shipwright/design/YYYY-MM-DD-<topic>.md`
12. Transition: "Run `/sw-plan docs/shipwright/design/…` to begin implementation planning."

### Happy Path — On-Demand Mode

1. User runs `/sw-design "rethink the checkout flow"` (no upstream doc)
2. Skill notes absence of discovery doc, proceeds with what the user provided
3. Asks one clarifying question if the scope is unclear — then runs the full strategist workflow
4. Output is a standalone design strategy artifact, useful with or without a plan following it

### Error Paths

1. **Discovery doc provided but not found**: warn "Could not read upstream doc at [path]. Proceeding without it — provide context in your description if needed." Never block.
2. **Scope too vague even after one clarifying question**: proceed with stated assumptions, mark them explicitly in the artifact. Don't loop on clarification.

## Feasibility Assessment

**Complexity:** Moderate

**Blockers:** None — new skill + role upgrade, no new infrastructure.

**Unknowns:**
- **Mermaid trigger threshold**: when exactly should a Mermaid diagram replace text? Resolution: use Mermaid for IA with 3+ levels of hierarchy, and for decision flows with 3+ branches. Text otherwise.
- **Role upgrade vs. new role**: should the existing `ux-designer.md` be upgraded in place, or should a new `ux-design-strategist.md` be created? Resolution: upgrade in place — the filename stays `ux-designer.md` to avoid breaking the existing markdown link in `sw-discover`; the internal name, voice, and capabilities are upgraded. The Architect model (support in discover, lead in plan) applies here: UX is support in discover, lead in design.

## Phase 2 Candidates

- **Design critique agent** — a dedicated adversarial UX agent (parallel to the review agents in sw-ship) that examines a design artifact for confusion points, accessibility failures, and IA inconsistencies
- **`sw-design --iterate`** — re-run the design phase on an existing artifact to refine it after feedback
- **Design ↔ Build feedback loop** — after sw-build, flag when implementation diverges from the design artifact's interaction principles
